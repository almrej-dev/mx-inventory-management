  <action>
    1. Create `src/middleware.ts` -- Next.js middleware that:
       - Calls `updateSession(request)` from lib/supabase/middleware.ts to refresh auth tokens
       - After session refresh, checks if user is authenticated via supabase.auth.getUser()
       - Redirects unauthenticated users to /login (except for /login route itself)
       - Redirects authenticated users away from /login to /
       - For /users route: reads JWT custom claim `user_role`, redirects non-admin users to / with an error
       - Matcher config: `["/((?!_next/static|_next/image|favicon.ico|api).*)"]`

    2. Create `src/lib/auth.ts` with `requireRole(minimumRole: AppRole)` function:
       - Creates server Supabase client
       - Calls supabase.auth.getUser() to verify authentication (NOT getSession -- getUser validates the JWT)
       - Gets session, decodes JWT with jwt-decode to read user_role custom claim
       - Compares against role hierarchy: viewer=1, staff=2, admin=3
       - Throws "Unauthorized" if insufficient role
       - Returns { user, role } on success
       - See RESEARCH.md code example for exact implementation

    3. Create `src/app/(auth)/layout.tsx` -- minimal layout with centered content, no sidebar/navigation. Just the root html structure wrapper for auth pages.

    4. Create `src/app/(auth)/login/page.tsx` -- Login page:
       - Email input, password input, submit button using shadcn/ui components (Input, Button, Card, Label)
       - Client component ("use client") that uses Supabase browser client
       - On submit: calls supabase.auth.signInWithPassword({ email, password })
       - On success: router.push("/") with router.refresh() to update server state
       - On error: shows error message below form (e.g., "Invalid login credentials")
       - Simple, clean card layout centered on page
       - NO registration form (admin creates users)

    5. Create `src/app/(dashboard)/layout.tsx` -- Dashboard layout:
       - Server component that reads user session
       - Renders sidebar (left) + main content area (right)
       - Passes user info and role to header component
       - Standard responsive layout: sidebar collapses on mobile

    6. Create `src/app/(dashboard)/page.tsx` -- Dashboard home page:
       - Simple welcome message with user name
       - Placeholder cards for future dashboard widgets (Phase 4)
       - Shows current role badge

    7. Create `src/components/layout/sidebar.tsx` -- Navigation sidebar:
       - Links: Dashboard (/), Items (/items), Stock Receiving (/stock/receiving), Users (/users -- admin only)
       - Use RoleGate to conditionally show admin-only links
       - Active link highlighting based on pathname
       - Use shadcn/ui components (Button with variant="ghost", icons optional)

    8. Create `src/components/layout/header.tsx` -- Top header:
       - Shows current page title
       - User menu (right side) with user email/name and Sign Out button
       - Sign out calls supabase.auth.signOut() then redirects to /login

    9. Create `src/components/layout/role-gate.tsx` -- Conditional render component:
       - Props: allowedRoles: AppRole[], children, fallback (optional)
       - Reads user role from context or prop
       - Renders children only if user has an allowed role
       - Otherwise renders fallback or nothing

    10. Create `src/schemas/user.ts` -- Zod schema for user creation:
        - email: z.string().email()
        - password: z.string().min(8, "Password must be at least 8 characters")
        - fullName: z.string().min(1, "Name is required")
        - role: z.enum(["admin", "staff", "viewer"])

    11. Create `src/actions/users.ts` -- Server actions for user management:
        - `createUser(rawData: unknown)`: Calls requireRole("admin"), validates with userSchema, calls supabase.auth.admin.createUser() using a service-role Supabase client (NOT the cookie-based one -- create a separate client with SUPABASE_SERVICE_ROLE_KEY), inserts user_roles record via raw SQL through Supabase client (supabase.rpc or supabase.from), creates Profile record via Prisma. Wraps in try/catch to clean up on partial failure.
        - `listUsers()`: Calls requireRole("admin"), queries Supabase auth.admin.listUsers() and joins with user_roles data.
        - `deleteUser(userId: string)`: Calls requireRole("admin"), deletes auth user via admin API, cascading deletes handle user_roles.

    12. Create `src/components/users/user-form.tsx` -- User creation form:
        - Client component with React Hook Form + Zod resolver
        - Fields: full name, email, password, role (select dropdown)
        - Submit calls createUser server action
        - Shows success/error feedback

    13. Create `src/app/(dashboard)/users/page.tsx` -- User management page:
        - Server component that calls listUsers() to get current users
        - Renders a table of users (name, email, role)
        - Button to open user creation form (dialog or inline)
        - Delete button per user row (with confirmation)

    14. Create a SQL migration file at `supabase/migrations/00001_rbac_setup.sql` containing:
        - CREATE TYPE public.app_role AS ENUM ('admin', 'staff', 'viewer')
        - CREATE TABLE public.user_roles (id bigint generated by default as identity primary key, user_id uuid references auth.users on delete cascade not null, role app_role not null, unique(user_id, role))
        - The custom_access_token_hook function (see RESEARCH.md Pattern 1)
        - The authorize() function (see RESEARCH.md Pattern 1)
        - RLS policies for items table (select: viewer+, insert/update: staff+, delete: admin only via soft-delete)
        - RLS policies for inventory_transactions (select: viewer+, insert: staff+)
        - RLS policies for profiles (select: viewer+, update: own profile only)
        - RLS policies for user_roles (select: admin only, insert/update/delete: admin only)
        - Grant execute on custom_access_token_hook to supabase_auth_admin
        - Revoke execute from public, anon, authenticated on custom_access_token_hook

        NOTE: This SQL must be run manually in Supabase SQL Editor or via supabase CLI before the app works. Include a comment at the top of the file explaining this. The executor should create this file AND include instructions in the SUMMARY for the user to run it.

    IMPORTANT: Use supabase.auth.getUser() (NOT getSession()) for auth verification -- getUser() validates the JWT server-side while getSession() only reads the unvalidated cookie.
